The fastest path from zero to a running LangSmith instance on EKS. Plan for 30 to 40 minutes end to end on a clean account.

<Tip>
**First time?** Run these commands in order. Each step builds on the previous. For configuration details and advanced options, jump to [Pass 1](/langsmith/self-host-terraform-aws-pass-1-infrastructure) and [Pass 2](/langsmith/self-host-terraform-aws-pass-2-langsmith).
</Tip>

Before starting, complete the [AWS prerequisites](/langsmith/self-host-terraform-aws-prerequisites) (tools, IAM permissions, license key, domain).

## Rapid path

```bash
# 1. Clone the public modules
git clone https://github.com/langchain-ai/terraform.git
cd terraform/modules/aws

# 2. Generate terraform.tfvars interactively (Enter accepts current values)
make quickstart

# 3. Load secrets into SSM Parameter Store
#    Must be sourced, not executed
source infra/scripts/setup-env.sh

# 4. Provision infrastructure (~20 to 25 min)
make init
make plan
make apply

# 5. Configure kubectl
make kubeconfig
kubectl get nodes

# 6. Deploy LangSmith via Helm (~5 to 10 min)
make init-values
make deploy

# 7. Confirm
kubectl get pods -n langsmith
kubectl get ingress -n langsmith
```

## Chained Pass 1 + Pass 2

After running `make quickstart` and sourcing `setup-env.sh`, chain both passes in one command:

```bash
make quickdeploy          # interactive — prompts before terraform apply
make quickdeploy-auto     # non-interactive — auto-approves terraform
```

`make quickdeploy` runs `terraform apply` → `kubeconfig` → `init-values` → `helm deploy` in sequence. If any step fails, the command exits with instructions for resuming from that step.

## What gets deployed

| Resource | Purpose |
|---|---|
| VPC + subnets + NAT | Private network for cluster and managed services |
| EKS cluster + node groups | Kubernetes compute |
| RDS PostgreSQL | LangSmith operational data |
| ElastiCache Redis | Queue and cache |
| S3 bucket + VPC endpoint | Trace payload blob storage |
| ALB + listeners | Public ingress with TLS |
| SSM Parameter Store entries | Application secrets, synced into the cluster by External Secrets Operator |
| IRSA roles + IAM policies | Per-service AWS access |
| KEDA, cert-manager, ESO | Bootstrap workloads installed during Pass 1 |
